# 项目背景

本文档说明整个项目的业务范围、前后端组成，以及当前 Python / Go 后端之间的关系。

## 1. 项目整体功能

这是一个找房与发房项目，当前至少包含三类核心业务场景：

### 1.1 小程序用户侧

主要面向租客用户，核心功能包括：

- 登录与身份认证
- 浏览房源
- 查看房源详情
- 收藏、历史记录
- AI 找房 / 对话找房

### 1.2 发房系统

主要面向房东或录入人员，核心功能包括：

- 录入房源主数据
- 维护集中式和分散式房源
- 逐步形成可发布的房源业务对象
- 后续产出展示层数据

### 1.3 后台管理系统

主要面向运营和管理人员，核心功能包括：

- 管理主数据
- 查看和维护房源
- 后续管理发布与展示数据
- 处理运营、审核、管理侧动作

## 2. 当前前端组成

当前可以明确有三类前端：

### 2.1 小程序前端

面向租客用户。

它主要依赖：

- 登录鉴权
- 房源搜索和详情
- AI 找房
- 收藏与历史

### 2.2 发房前端

面向房东或录入人员。

它主要依赖：

- `publish` 域接口
- HMD 主数据录入
- 后续 HPD 发布与展示准备

### 2.3 后台管理前端

面向运营和管理人员。

它不是发房前端的一部分，也不是小程序前端的一部分。

它主要依赖：

- 管理侧 API
- HMD / HPD 的管理与维护能力

## 4. Python 后端与 Go 后端的关系

两者关系可以概括为：

- Python 后端是当前可运行参考
- Go 后端是迁移目标实现

迁移原则是：

- 切换语言
- 保留业务规则和业务行为
- 不主动引入兼容设计
- Go 项目按当前清晰分层重新实现
- 不迁移 Python 项目的技术边界、临时写法和历史包袱

也就是说，Python 项目是业务行为参考，不是 Go 项目的技术架构模板。

## 5. 各前端与各后端的关系

## 5.1 当前关系

当前可以这样理解：

- 小程序前端：
  - 已有 Python 版业务参考
  - Go 版已先跑通 auth 等基础链路
- 发房前端：
  - 接下来重点由 Go 后端继续实现
  - 当前接口和架构仍在设计与落地阶段
- 后台管理前端：
  - 目前还没有进入优先实现阶段
  - 后续会基于 Go 后端继续接

## 5.2 后端职责划分

当前更准确的理解不是“一个前端对应一个后端”，而是：

- Python 后端：历史可运行版本与迁移参考
- Go 后端：正在逐步接管各个前端需要的业务能力

## 6. 当前迁移重点

当前迁移顺序不是三套前端一起推进，而是：

1. 先把 Go 后端基础能力打稳
2. 先跑通 auth
3. 先完成 HMD repository
4. 再继续推进发房系统 `publish` 域
5. 再继续推进 HPD
6. 最后再补后台管理侧能力

## 7. 当前结论

这个项目不是单前端、单后端的简单结构。

它更接近：

- 一个租客侧小程序
- 一个发房前端
- 一个后台管理前端
- 一个历史 Python 后端参考实现
- 一个正在接管业务的 Go 后端

而当前迁移阶段，最重要的后续主线是：

> 继续用 Go 把发房系统 `publish` 域做起来，并为后续 HPD 和后台管理能力留出正确架构。
