# Admin

后台管理系统是平台员工工作台，用于审核、运营、分发、跟进和账号权限管理。

本文档只描述产品需求和业务边界，不写 Go / Vue 实现细节。接口契约、数据库字段和实现计划分别维护在 `api/`、`schema/`、`backend/`、`workstreams/`。

## 产品定位

三端分工：

- `miniapp`：租客找房、聊天、收藏、预约。
- `publish`：房东录入和维护自己的房源。
- `admin`：平台员工管理平台流程。

后台管理不是发房系统增强版，也不是房东视角页面集合。后台可以复用底层后端能力，但不能直接复用 miniapp / publish 的端侧业务服务。

## 第一阶段目标

第一阶段建立平台最小管理闭环：

1. 平台员工可以登录后台。
2. 平台可以管理员工、角色和权限。
3. 平台可以管理发房方账号。
4. 平台可以查看和筛选全量房源。
5. 平台可以处理房源上架审核。

第一阶段的成功标准：

- 新员工可以由后台创建并登录。
- 员工菜单和操作按钮可以根据权限控制。
- 发房方账号可以由平台创建、查看、编辑和禁用。
- 运营可以按核心筛选项查看全量房源。
- 上架审核可以形成审核任务，并由有权限员工通过或驳回。

## 第一阶段不做

以下能力不进入第一阶段：

- 客户消息完整会话工作台。
- 预约看房 CRM 化跟进。
- 广告、弹窗、推送等营销系统。
- AI 画像、自动推荐、复杂分单策略运营配置。
- 房源信息变更审核。
- 后台直接修改房源服务信息或联系人信息。
- 员工忘记密码的短信验证码自助重置。
- 历史后台、旧前端、旧数据库兼容。

## 用户与权限模型

后台使用员工身份：

- 登录主体是 `staff`。
- 后台 session 固定为 `terminal=admin`。
- 后台员工、房东、C 端用户是不同主体，不能混用身份 ID。

第一阶段角色：

- `super_admin`：超级管理员，拥有全部权限。
- `ops_admin`：运营管理员，负责发房方、房源、审核等运营动作。
- `viewer`：只读账号，只能查看。

第一阶段权限原则：

- 查看类接口需要对应 `*.view` 权限。
- 编辑、创建、禁用、审核类接口需要对应 `*.edit` 或更具体权限。
- 前端可以根据权限控制菜单和按钮，但后端必须再次校验。
- 员工不能通过请求参数提交自己的 `staff_id` 来伪造操作人，操作人必须来自 admin session。

已确认权限字典以数据库设计为准；进入 `launch_audit` 实现前，需要补齐审核模块权限点，例如查看审核和处理审核。

## 模块需求

### 1. 登录与会话

范围：

- 员工手机号密码登录。
- 恢复当前登录态。
- 退出登录。

业务规则：

- 登录手机号来自后台员工主体。
- 密码只存认证表中的 bcrypt hash，不存明文。
- 登录成功后返回 opaque session token，不使用 JWT。
- 登录失败不暴露“手机号不存在 / 密码错误 / 账号禁用”的具体原因。
- 禁用员工不能登录。
- session 恢复时必须重新读取员工状态和权限，不能只信 Redis 里的旧权限快照。
- 退出登录只失效当前 token。

暂不做：

- 忘记密码短信验证码重置。
- 邮箱登录。
- 多因素认证。

验收标准：

- 正确手机号密码可以登录并拿到角色和权限。
- 错误手机号或密码返回统一中文错误提示。
- 有效 token 可以恢复当前员工资料和权限。
- 退出后当前 token 不可继续恢复 session。

### 2. 员工管理

范围：

- 员工列表。
- 员工详情。
- 创建员工。
- 编辑员工资料。
- 禁用员工。

业务规则：

- 员工手机号全局唯一。
- 创建员工时必须创建员工主体和密码认证信息。
- 员工姓名、手机号、初始密码为创建员工的最低必填信息。
- 员工角色可以为空，但无角色员工只能登录，不能访问需要权限的业务功能。
- 员工手机号第一阶段不支持修改；如确需变更，后续单独设计。
- 禁用员工不物理删除。
- 禁用员工后，该员工不能继续登录；是否立即踢掉已有 session 在实现 `staff/disable` 前单独设计 session 失效方案。
- 创建、编辑、禁用员工都必须记录操作人。

页面需要：

- 员工列表支持按关键词、手机号、角色、状态筛选。
- 员工列表展示姓名、手机号、角色、状态、创建时间、最近登录时间、最近登录 IP。
- 员工详情展示基础资料、角色列表、创建人、登录信息。

验收标准：

- 有权限员工可以创建新员工并分配角色。
- 重复手机号不能创建。
- 无权限员工不能创建、编辑或禁用员工。
- 禁用员工不能再登录。

### 3. 角色权限

详细需求：

- [角色权限模块需求文档](../workstreams/admin-web/active/role-requirements.md)

范围：

- 角色列表。
- 角色详情。
- 创建角色。
- 编辑角色权限。

业务规则：

- 角色编码系统唯一。
- 系统内置角色不能被删除。
- `super_admin` 是系统内置角色，拥有全部权限。
- 角色权限来自后端维护的权限字典，不允许前端提交任意权限码后直接生效。
- 编辑角色权限后，新 session 恢复时必须能读取到最新权限。
- 第一阶段不做角色删除，避免误删导致员工权限断链。

页面需要：

- 角色列表展示角色名称、角色编码、权限摘要、创建时间。
- 角色详情展示完整权限树或权限码列表。

验收标准：

- 有权限员工可以创建角色并配置权限。
- 非法权限码不能保存。
- 被分配角色的员工能在 session 中拿到对应权限。

### 4. 发房方管理

详细需求：

- [发房方管理模块需求文档](../workstreams/admin-web/active/provider-requirements.md)

范围：

- 发房方列表。
- 发房方详情。
- 创建发房方。
- 编辑发房方。
- 禁用发房方。

业务规则：

- 发房方接口层 `provider_id` 等于房东主体 ID。
- 第一阶段只实现发房方主体和认证，即 `hs_lld_landlord + hs_lld_auth`。
- `hs_lld_profile` 已有 schema 位置，但第一阶段不进入实现。
- 发房方手机号唯一。
- 创建发房方时需要创建登录认证信息。
- 禁用发房方不物理删除。
- 禁用发房方后，发房端不能继续登录。
- 后台不能把发房方资料硬塞进 C 端用户或房源主数据。

页面需要：

- 当前账号阶段发房方列表支持按手机号、状态筛选。
- 当前账号阶段发房方详情展示基础账号信息和登录摘要。
- 城市、名称、图片、品牌资料等依赖 `hs_lld_profile`，后续进入资料阶段再实现。

验收标准：

- 平台可以创建可登录 publish 的发房方账号。
- 重复手机号不能创建。
- 禁用后发房方不能登录 publish。
- 后台创建人和最近更新人可以追踪。

### 5. 房源管理总览

范围：

- 平台视角项目/小区列表。
- 楼栋列表和房间列表查看。
- 房间详情查看。
- 房源导出接口名保留，具体导出方案实现前单独确认。

业务规则：

- admin 是全量运营视角，不走 publish 的房东 owner scope。
- 房源管理第一阶段只做查看，不做后台直接编辑房源服务信息或联系人信息。
- 房源管理读接口以 HPD 后台读模型为基础，不直接复用 miniapp `house` 路由。
- 小程序可见性仍由 HPD 发布状态和 HMD 房态共同决定。

页面需要：

- 支持按来源、发房方、项目、资产类型、城市、区域、审核状态、上下架状态筛选。
- 首页展示项目/小区名称、发房方、位置、楼栋数、房间数、更新时间。
- 房间列表展示房源标题、发房方、位置、价格、房态、发布状态、审核状态、更新时间。
- 详情展示发房方信息、项目/楼栋/小区、房型/房间、当前状态、审核状态。

验收标准：

- 运营可以筛选并查看全量房源。
- 无权限员工不能访问房源列表和详情。
- 后台查看不改变房源状态。

### 6. 房源上架审核

范围：

- 上架审核列表。
- 上架审核详情。
- 审核通过。
- 审核驳回。

业务规则：

- 审核必须以审核单为中心，不能直接对 HMD 主对象做裸审核。
- 第一阶段只做上架审核，不做房源信息变更审核。
- 审核任务状态至少包括待审核、审核通过、审核驳回。
- 审核通过后，更新对应发布对象的发布状态，并刷新展示层可见性。
- 审核驳回后，保留驳回原因，发布对象不进入可展示状态。
- 审核人必须来自 admin session。
- 审核动作必须记录审核时间、审核人、审核备注。

页面需要：

- 审核列表支持按来源、发房方、资产类型、状态筛选。
- 审核详情展示审核单基础信息、当前生效内容、待审核内容、差异摘要、审核备注输入区。

验收标准：

- publish 端提交上架后，后台能看到待审核任务。
- 有权限员工可以通过或驳回。
- 审核结果能稳定落到审核单和发布状态。
- 已处理审核单不能重复处理。

## 操作审计

第一阶段至少要求以下动作有可追踪信息：

- 员工登录成功 / 失败。
- 创建、编辑、禁用员工。
- 创建、编辑角色权限。
- 创建、编辑、禁用发房方。
- 审核通过 / 驳回。

审计信息至少包括：

- 操作人。
- 操作时间。
- 操作对象。
- 操作结果。
- 失败原因或备注。

登录审计已有独立集合；其它操作审计是否统一建集合，在进入对应模块实现前确认。

## 错误提示

后台管理接口错误提示必须是完整中文句子：

- 参数错误应提示用户如何修正。
- 权限不足应明确提示当前账号无权操作。
- 登录失败不暴露账号枚举信息。
- 服务端异常不返回底层库错误。

示例：

- `请输入手机号和密码`
- `手机号或密码错误，请重新输入`
- `当前账号无权执行该操作`
- `员工手机号已存在，请更换后重试`
- `当前记录状态已变化，请刷新后重试`

## 第二阶段候选

- 预约看房。
- 客户消息。
- 营销管理。
- 员工短信验证码自助找回密码。
- 房源信息变更审核。
- 更完整的操作审计中心。
