# 数据库设计总览（V3 草案）

## 1. 文档目标

本文档用于描述 AI 找房租房项目的 MongoDB 数据模型分层与集合划分。

当前项目已不再是单纯的小程序找房 MVP，而是覆盖以下系统边界：

- 小程序前端
- 后端服务（Go / Python）
- 后台管理端
- MongoDB + Redis
- AI 聊天与推荐链路

因此，数据库设计不再适合继续围绕单一 `house` 集合扩展字段，而应按业务域拆分为多组集合。

本文档目录中的子文件按模块组织，分别说明各模块下集合的职责与核心字段。

## 2. 设计原则

### 2.1 主数据与流程数据分离

- 主数据用于描述相对稳定的业务对象，例如项目、楼栋、小区、房型、房间。
- 流程数据用于描述变化中的业务过程，例如审核、预约、跟进、推送、AI 推荐记录。

### 2.2 后台维护模型与前台展示模型分离

- 后台维护模型承载完整业务关系。
- 前台展示模型只承载对外可展示、可检索、可推荐的房源数据。

### 2.3 C 端用户与 B 端员工分离

- 小程序用户与后台员工承担完全不同的业务职责。
- 两者可在认证层共享部分能力，但不应混用同一份业务档案结构。

### 2.4 先保证边界清晰，再细化字段

当前版本先确定集合边界与职责，再逐步补充字段与索引设计。

## 3. 模块划分

MongoDB 建议按以下八个模块组织集合：

1. 账号与身份模块
2. 房源主数据模块
3. 房源发布与展示模块
4. 房源审核与变更模块
5. 客户与业务跟进模块
6. 聊天与 AI 模块
7. 运营与营销模块
8. 用户行为模块

## 5. 关系说明（高层）

### 5.1 房源主线

房源相关数据建议分为三层：

1. 房源主数据  
   用于表达项目、楼栋、小区、房型、房间等真实业务对象。

2. 房源发布与展示  
   用于表达哪些房源当前可对外展示，以及前台搜索和详情查询使用的数据视图。

3. 房源审核与变更  
   用于表达上架审核、信息变更审核、发布状态流转等过程数据。

### 5.2 用户主线

用户相关数据建议分为三层：

1. C 端用户身份与档案
2. B 端员工、角色与权限
3. 用户行为与业务动作

### 5.3 AI 与业务闭环

AI 聊天不是独立系统，而是以下流程的入口或加速器：

- 找房需求识别
- 推荐记录生成
- 客服分配
- 私人定制线索生成
- 委托出租线索生成

相关数据需在聊天模块和客户跟进模块之间形成可追踪记录。
