# 智小窝找房数据库设计文档 (V4.0)

## 一、 数据库设计思路
本设计旨在通过数据结构定义业务边界，核心逻辑如下：
- **权属绑定**：通过 `owner_id` 建立房源与房东的 1:N 关联，为房东后台的增删改查提供权限隔离基础。
- **状态机控制**：引入 `status` 字段作为业务总开关。在代码实现上，AI 检索逻辑将强制挂载 `status=1` 约束，确保业务合规性。
- **输入标准化**：预设 `type`（户型）和 `tags`（标签）字典，规范化房东输入，同时提升 NLU 模块对房源特征的匹配精度。
- **工程高性能**：针对高频的"房东管理"与"AI 检索"场景，预设复合索引策略，避免全表扫描引发的性能瓶颈。

## 二、 集合定义 (Collections)

### 2.1 房源详情表 (`hs_hs_house`)
**职责**：持久化存储房源物理属性、价格、归属及实时状态。

| 字段名 | 类型 | 必填 | 规范化取值/说明 |
| :--- | :--- | :--- | :--- |
| `_id` | String | 是 | MongoDB 原生主键|
| `owner_id` | String | 是 | 房东 ID（关联用户表 `user_id`） |
| `area` | String | 是 | 行政区（例：南山, 福田, 宝安, 罗湖） |
| `location` | String | 是 | 小区/街道具体位置 |
| `type` | String | 是 | **标准字典**：`开间`, `一室一厅`, `二室一厅`, `三室两厅`, `合租单间` |
| `price` | Int | 是 | 月租金（整数） |
| `tags` | List[Str] | 否 | **标准标签集**（详见下文 2.1.1） |
| `status` | Int | 是 | **状态机**：`1`:待租, `2`:已租, `0`:下架/逻辑删除 |
| `images` | List[Str] | 否 | 房源照片 URL 列表 |
| `created_at` | Int64 | 是 | 创建时间戳（秒级） |
| `updated_at` | Int64 | 是 | 最后修改时间戳（秒级） |

#### 2.1.1 `tags` 标准化字典设计
房东发布房源时，仅允许或优先推荐以下标签，以便 AI 进行特征匹配：
- **配套**：`家电齐全`, `拎包入住`, `有电梯`, `私密独卫`, `可做饭`
- **地理**：`近地铁`, `步行可达`, `核心商圈`, `安静内街`
- **政策**：`押一付一`, `随时看房`, `免中介费`, `可养宠物`

### 2.2 用户档案表 (`hs_usr_user`)

> **复用会话模块共享表，定义见 `db_session_ds.md` § 2.1。**
> 房源模块通过 `hs_hs_house.owner_id` → `hs_usr_user.user_id` 建立关联，`role` 字段用于区分房东（`landlord`）与租客（`tenant`）。

## 三、 索引配置 (Indexing)

为确保生产环境下查询效率，必须在 MongoDB 层面预设以下索引：

**`hs_hs_house` 房源表**
1. **房东管理复合索引**：`{"owner_id": 1, "status": 1}`
   - 用于房东后台高频查询"我的有效房源"。（`_id` 为 MongoDB 原生主键，自带唯一性，无需额外建索引）
2. **AI 检索复合索引**：`{"area": 1, "price": 1, "status": 1}`
   - 用于支撑 AI 客服多维过滤，强制包含 `status` 以实现业务隔离。

**`hs_usr_user` 用户表**
> 索引由会话模块统一管理，详见 `db_session_ds.md` § 2.1，此处不重复定义。

---

## 四、 工程约束
- **写操作校验**：所有涉及 `update` / `delete` 的 API 必须强制校验 `request.user_id == db.owner_id`。
- **数据一致性**：房源变更采用 **Write-Through** 模式，直接持久化至 MongoDB，不经过中间缓存层，确保 AI 检索结果即时更新。
