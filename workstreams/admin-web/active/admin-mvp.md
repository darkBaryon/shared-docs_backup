# 管理后台 Admin MVP 开发文档

本文档是管理后台第一阶段的统一开发入口。产品、页面、接口、后端模块和实施顺序都从这里对齐。

配套文档：

- 后端详细计划：`admin-backend-plan.md`
- 前端详细计划：`admin-frontend-plan.md`

当前补充约束：

- schema 还没构建完，当前计划不能直接进入开发。
- 高风险问题先通过 schema / 状态流 / 主键锚点收口，再进入代码阶段。

## 1. 产品定位

管理后台是平台员工工作台，不是房东发房端。

三端分工：

- `miniapp`：租客找房、聊天、收藏、预约
- `publish`：房东录入和维护自己的房源
- `admin`：平台员工审核、运营、分发、跟进、账号与权限管理

## 2. 第一阶段目标

先建立平台主链最小闭环：

1. 员工可以登录后台
2. 平台可以管理员工、角色、权限
3. 平台可以管理发房方
4. 平台可以查看和筛选全量房源
5. 平台可以处理房源上架审核

## 3. 第一阶段不做

以下功能不进入第一阶段实现：

- 客户消息完整会话工作台
- 预约看房 CRM 化跟进
- 广告、弹窗、推送等营销系统全量实现
- 所有 AI 画像、自动推荐、复杂分单策略的全量运营端实现
- 房源信息变更审核
- 后台直接修改房源服务信息 / 联系人信息

这些需求先保留在产品文档，不作为第一阶段交付范围。

## 4. 信息架构

管理后台第一阶段一级导航建议固定为：

## 6. 后端模块清单

第一阶段后端建议拆为：

- `admin/auth`
- `admin/house`
- `admin/launch_audit`
- `admin/provider`
- `admin/staff`
- `admin/role`

依赖关系：

```text
handler/v1/admin/{module}
  -> service/admin/{module}
    -> domain/*
    -> repository/*
```

## 7. 数据模型方向

第一阶段需要明确的数据模型：

- 员工、角色、权限
- 发房方业务资料
- 房源上架审核单

其中：

- HMD / HPD 主数据继续沿用现有体系
- 管理后台不直接复用 `publish` service
- 审核必须有“审核单”概念，不直接对 HMD 对象裸审核

## 8. 前端约定

- 管理后台是典型运营控制台，不采用 `publish` 的录入工作台交互模型。
- 页面以高密度列表、筛选、批量操作、详情抽屉为核心。
- 不做营销站式布局，不做大面积卡片化首页。
- 权限相关页面优先保证清晰、可审计、可维护。

## 10. 分阶段实施顺序

### Phase 0：文档与 schema 口径

目标：

- 固化 admin 产品范围
- 固化 admin API 第一批模块
- 固化 admin backend / frontend 文件架构
- 明确第一批 schema 缺口
- 明确 provider 主键锚点
- 明确 launch audit 状态流

交付：

- `product/admin.md`
- `frontend/admin.md`
- `api/admin.md`
- `backend/admin.md`
- `workstreams/admin-web/active/*`

### Phase 1：认证与组织体系

目标：

1. 员工登录
2. session 恢复
3. 员工管理
4. 角色权限管理

范围：

- `admin_auth`
- `staff`
- `role`

完成标志：

- 员工可登录后台
- 可创建员工
- 可分配角色和权限
- 后台页面能根据权限控制菜单和操作按钮

### Phase 2：发房方管理

目标：

1. 发房方列表
2. 发房方详情
3. 发房方创建与编辑
4. 发房方禁用

范围：

- `provider`

完成标志：

- 平台可以独立维护发房方资料
- 发房方账号状态可控
- 发房方资料和 publish 房源能建立稳定关联

### Phase 3：房源管理总览

目标：

1. 平台视角全量房源列表
2. 房源详情查看
3. 审核状态查看
4. 导出

范围：

- `house`

完成标志：

- 运营可以按来源、发房方、项目、类型筛选全量房源
- 可以稳定查看房源当前状态和审核状态

### Phase 4：房源审核

目标：

1. 上架审核列表
2. 审核详情
3. 审核通过 / 驳回

范围：

- `launch_audit`

完成标志：

- publish 端需要审核的对象在后台可见
- 审核动作会稳定落到审核单和主数据

### Phase 5：二期评估

候选：

- 预约看房
- 客户消息
- 营销管理

## 11. 执行规则

1. 先完成一个 phase 的产品、接口、文件计划，再进入代码实现。
2. 每个 phase 都要同时维护：
   - API 契约
   - backend 文件计划
   - frontend 文件计划
3. 未写进文档的临时页面、临时接口、临时目录不允许直接进入实现。
4. 如果 phase 中途变更范围，先改本文档和对应详细计划，再改代码。
5. `auth` 是当前唯一保留 terminal 前缀的例外模块，因为 miniapp 已占用 `/auth/session`；其它 admin 模块遵守 `<module>/<action>`，不加 terminal 前缀。
